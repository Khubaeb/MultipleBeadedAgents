01 — Foundation

Purpose & mental model

Beads is a dependency-aware, version-controlled issue graph built for AI coding agents and the humans who supervise them.

What it is

A persistent, structured memory layer. Beads replaces fragile markdown plans with a real graph: issues have typed dependencies, blockers, hierarchy, and a full audit trail. The graph is version-controlled by Dolt, so multiple agents and humans can edit it concurrently without losing context.

Who uses it

Coding agents
Discover, claim, do, close, and remember.
Humans
Plan, triage, audit, override.
Orchestrators
Route work, enforce policies, run gates.
Multi-agent teams
Sync state across machines and processes.

Mental model

Think of Beads as a tiny database with a CLI front-end. You create issues, connect them with dependencies, let an agent drain the ready queue, and let the graph stay the single source of truth. Workflows are expressed as formulas that cook into instances — molecules or wisps.

Promise

An agent that comes back tomorrow can re-derive what it was doing by reading the graph. That is the durable-work contract.

02 — Foundation

Storage & truth

A local Dolt database is the source of truth. JSONL is a passive export. Cross-machine sync rides refs/dolt/data.

Two modes

ModeStorage pathConcurrencyDefault?
Embedded.beads/embeddeddolt/Single writer (file lock)Yes
Server.beads/dolt/Multiple writersNo

Three things inside the database

  • Issue tables — issues, labels, dependencies, events, comments. Synced.
  • Wisp tables — wisps, wisp_labels, wisp_dependencies, wisp_events, wisp_comments. Marked dolt_ignored. Never synced, never version-controlled.
  • Meta tables — schema_migrations, configuration.
Anti-pattern

Never treat .beads/issues.jsonl as the wire protocol. It is a throttled export for viewers and interchange. Cross-machine sync uses bd dolt push / bd dolt pull against refs/dolt/data on the git remote.

03 — Foundation

Install & repository setup

Install bd once on the machine, then initialise inside each project.

Install paths

MethodCommand
Homebrew (macOS/Linux)brew install beads
npm (any platform)npm install -g @beads/bd
Windowswinget install beads
Install scriptcurl -fsSL https://raw.githubusercontent.com/gastownhall/beads/main/scripts/install.sh | bash

Initialise a project

cd your-project
bd init

This creates .beads/, optionally writes AGENTS.md, and (if git origin is present) wires a Dolt remote named origin.

Init flags

FlagEffect
--stealthNo git ops; useful for non-git VCS or one-off work.
--serverUse an external dolt sql-server instead of embedded.
--prefix <pfx>Set the issue ID prefix (default bd).
--from-jsonlBootstrap from an existing issues.jsonl.
--contributorRoute planning issues to a separate namespace.

Agent integrations

Install the right integration for your tool. bd setup writes hooks and a workflow block into your tool’s instruction file:

bd setup codex     # Beads skill + AGENTS.md + Codex hooks
bd setup claude    # SessionStart/PreCompact hooks + minimal section
bd setup factory   # AGENTS.md marked section
bd setup cursor    # .cursor/rules/beads.mdc
bd setup gemini    # SessionStart/PreCompress hooks
bd setup mux       # AGENTS.md marked section
Verify

bd setup <recipe> --check confirms the integration is correctly installed.

05 — Daily use

Daily lifecycle

The four-beat loop every agent runs, plus the transitions that surround it.

1
create
2
claim
3
work
4
close

Create

bd create "Title" -p 1 -t bug --description "Detailed context"
bd create "Found issue" --deps discovered-from:bd-123
echo "Long description from stdin" | bd create "Title" --description=-

Claim & update

bd update bd-42 --claim             # atomic: assignee + in_progress
bd update bd-42 --priority 1
bd update bd-42 --status in_progress
bd update bd-42 --description "new text"

--claim is the optimistic-lock primitive. It only succeeds if the issue is unclaimed.

Close & reopen

bd close bd-42 --reason "Fixed in PR #142"
bd reopen bd-42
bd defer bd-42            # explicit postpone
bd supersede bd-42 bd-99  # replace with another

Comments & labels

bd comment bd-42 "Found root cause: race in token refresh"
bd label bd-42 backend auth
bd label bd-42 --remove wontfix
Agent warning

Do not use bd edit — it opens $EDITOR and is unusable for AI agents. Use bd update --description/--title/--design/--notes/--acceptance with stdin (--description=-) when the text contains backticks, exclamation marks, or nested quotes.

06 — Graph

Graph & ready work

The dependency graph is what makes Beads a work pump rather than a to-do list.

Blocking dep types

TypeSemanticsUse case
blocksB cannot start until A closes.Sequential work.
parent-childOpen parent keeps children unready; structural edge.Epic hierarchies.
conditional-blocksB runs only if A fails.Error-handling paths.
waits-forB waits for all of A’s dynamic children.Fanout gates.

Non-blocking dep types

related discovered-from caused-by tracks supersedes duplicates relates-to replies-to authored-by assigned-to approved-by attests validates delegated-from — graph annotations only. They do not affect bd ready.

Manipulation

bd dep add <child> <parent>                    # default: blocks
bd dep add <child> <parent> --type tracks
bd dep add <child> --blocked-by <parent>
bd dep issue-1 --blocks issue-2
bd dep remove <child> <parent>
bd dep rm <child> <parent>                      # alias

Ready & blocked

bd ready                       # no open blockers
bd ready --priority 1 --label backend
bd ready --assignee alice
bd ready --unassigned
bd ready --include-ephemeral   # include wisps
bd ready --gated               # also show post-gate resume
bd blocked                     # what's waiting and why
bd dep tree <id>              # visualise the graph
Hierarchy & parallelism

Children of an epic are parallel by default. Only explicit dependencies create sequence. Closing a parent does not cascade-close children.

07 — Discoverability

Find & understand work

Read commands, filters, and the audit trail.

Core reads

bd list                          # everything visible
bd list --status open --priority 0
bd list --assignee alice --label backend
bd show <id>                     # full detail + history
bd children <id>                 # parent → kids
bd show <id> --thread            # threaded view

Search & discovery

bd search "auth refresh"
bd search --status open "rate limit"
bd duplicates --auto-merge        # find and merge near-duplicates
bd orphans                       # children with no parent
bd stale                         # old open issues
bd lint                          # quality checks
bd stats                         # counts by status / type / priority

Reports

bd stats --json                  # machine-readable
bd list --json                   # one JSON object per issue
bd show <id> --json              # full audit trail in JSON
Agent tip

Always pass --json in agent scripts. The CLI is the stable API; do not parse the human-readable output.

08 — Collaboration

Collaboration & sync

Dolt rides refs/dolt/data on the git remote. Source Git and Dolt refs are separate.

Push & pull

bd dolt pull       # fetch + merge from remote
bd dolt push       # publish commits

When bd init runs in a repo with a git origin, Beads automatically wires a Dolt remote named origin so that bd dolt push / bd dolt pull flow over your existing git remote.

Conflicts

bd vc conflicts    # unresolved Dolt merge
bd vc resolve      # resolve interactively or with flags
bd vc commit       # commit pending merge state

Worktrees

Worktrees operate directly on the database — no special coordination needed. Use bd dolt push when each worktree is ready.

Multi-repo

Multiple repos can each have their own .beads/. To bridge them, use Dolt remotes (bd backup init /path, then bd backup sync) or one-time bootstrap via bd import from issues.jsonl.

Anti-patterns
  • Do not treat JSONL as the sync protocol.
  • Do not run bd import during normal operation.
  • Do not reach for third-party Dolt hosting before trying the default remote.
09 — Agent loop

Agent workflow & memory

bd prime is the single source of truth for operational context. bd remember persists project memory across sessions.

The canonical agent loop

1. bd prime                                    # inject context
2. bd ready --json                              # what can I do?
3. bd update <id> --claim --json               # claim it
4. work
5. bd close <id> --reason "..." --json         # done
6. (loop)

Memory & attribution

bd remember "Always use the secret loader, not env vars directly"
bd prime                  # later session reads it back
bd comment <id> "..." --actor claude-opus
bd create "..." --actor claude-opus

Recipe profiles

bd setup ships profiles for policy:

  • conservative (default) — track tasks with Beads; do not commit, push, or run Dolt remote sync without explicit approval.
  • minimal — short file that points to bd prime.
  • team-maintainer — only when the repository explicitly delegates session-close authority.
Note

The installed Beads block defaults to conservative git authority. Beads does not infer write permission just because a remote exists.

10 — Formulas

Formula system

A formula is a TOML (or legacy JSON) file describing a reusable workflow as a DAG of steps, variables, inheritance, composition, and advice.

File format

ExtensionStatus
.formula.tomlPreferred
.formula.jsonLegacy

Search-path priority (highest first)

  1. <beads-dir>/formulas/ — per-project override
  2. <checkout-root>/.beads/formulas/ — repo-shipped
  3. ~/.beads/formulas/ — user
  4. $GT_ROOT/.beads/formulas/ — orchestrator

Schema (top-level)

KeyRole
formulaName (also the proto ID).
description, versionMetadata.
typeworkflow | expansion | aspect | convoy.
extendsParent formula names — cycle-detected.
varsNamed variables with default/enum/pattern/required.
stepsArray of Step objects (see below).
composebond_points, expand, map, branch, gate, aspects.
adviceAspect advice rules (glob-based before/after/around).

Step fields

KeyMeaning
id, title, description, notesContent.
type, priority, labels, metadataIssue-shape metadata.
depends_on / needsSequential dependencies.
waits_forFanout gate dependency.
assigneeDefault assignee.
expand, expand_varsInline expansion of another formula.
conditionCook-time filter.
childrenNested steps.
gateRuntime gate (auto-creates a gate issue).
loopIteration (count / range / until+max).
on_completeDynamic bonding on step completion.

Variables

Variables use {{name}} placeholders in any string field. Provide them at pour/wisp time with --var key=value.

[vars.feature_name]
description = "Name of the feature"
required = true

[vars.priority]
default = "2"
enum = ["0", "1", "2", "3", "4"]

Cook-time conditions

[[steps]]
id = "docs"
title = "Write docs"
condition = "{{needs_docs}}"   # include iff needs_docs is truthy
condition = "!{{needs_docs}}"  # negated
condition = "{{region}} == eu"

Discover & convert

bd formula list
bd formula list --type workflow
bd formula show feature-workflow
bd formula convert <name>        # JSON → TOML
bd formula convert --all
Pipeline order (fixed)

control flow → advice → inline expansions → expansions → aspects. Composition depth is bounded at 5.

11 — Workflow instances

Workflow instances

Three flavours of runnable work: protos (templates), molecules (liquid, durable), and wisps (vapor, ephemeral).

Proto

A proto is an issue with the template label. By default it is ephemeral — cooked in memory at use time, nothing written to the database.

# Modern path: cook inline, no DB row
bd mol pour feature-workflow --var feature_name="auth flow"
bd mol wisp feature-workflow --var feature_name="auth flow"

# Legacy path: persist the proto (avoid in new work)
bd cook feature-workflow --persist --force

Molecule (liquid)

A molecule is an instantiated DAG: a parent epic plus children connected by typed dependencies.

  • Root type: molecule.
  • Steps connected by blocks (sequential), conditional-blocks (error path), waits-for (fanout).
  • Syncs via bd dolt push.
  • IDs: hierarchical root.step.
bd mol pour feature-workflow --var feature_name="auth flow"
bd mol show <mol-id>

Wisp (vapor)

A wisp is an ephemeral issue with Ephemeral=true. It lives in dolt_ignored tables and is never pushed, pulled, or synced.

PropertyLiquid (issue)Vapor (wisp)
Tablesissues, labels, dependencieswisps, wisp_labels, wisp_dependencies
ID formatbd-a1b2bd-wisp-x7k
Cross-machine?Yes (via bd dolt push/pull)No
GC candidate?Only via bd compactYes (bd mol wisp gc)

Promote

Move a wisp into the durable graph when the work has lasting value:

bd promote <wisp-id>

Preserves the ID and retargets inbound dependencies. Runs in one transaction.

Wisp types

Categorical: heartbeat, ping, patrol, gc_report, recovery, error, escalation. Infra types (agent, role, message) are protected from GC.

12 — Advanced control

Advanced control: bonds & gates

Bonds connect work graphs. Gates block steps until an external condition is true.

Bonds

A bond is a typed dependency between two protos, a proto and a molecule, or two molecules.

TypeEffectBacking dep
sequentialB runs after A.blocks
parallelB runs alongside A.parent-child
conditionalB runs only if A fails.conditional-blocks
 
bd mol bond A B                   # default sequential
bd mol bond A B --type parallel
bd mol bond A B --type conditional
bd mol bond A B --ref arm-{{name}} --var name=ace

The --ref flag enables the Christmas Ornament pattern: dynamic per-item children with deterministic hierarchical IDs.

Bond points

A formula can declare named attachment sites via compose.bond_points. At v1.1.0 these are authoring metadata only — there is no CLI flag to address them from bd mol bond. Wire by step ID instead.

Gates

A gate is an issue of type gate that blocks another issue until a condition resolves.

TypeResolved byv1.1.0 status
humanbd close <gate-id>Supported
timerbd gate check (auto-close on expiry)Supported
gh:runbd gate check shells out to gh CLISupported (workstation-bound)
gh:prSame as gh:runSupported (workstation-bound)
beadMulti-rig routing removedDisabled
mail—Documented, not implemented in gate check

Creating & managing gates

bd gate create --type=human --blocks <step-id>
bd gate create --type=timer --blocks <step-id> --timeout 1h
bd gate check                    # evaluate all
bd gate check --limit 100        # default limit
bd gate check --dry-run
bd gate add-waiter <gate-id> <agent-addr>
bd gate resolve <gate-id>      # alias for bd close
bd ready --gated                 # show molecules with newly-ready steps
v1.1.0 caveat

gh:run and gh:pr gate types require the gh CLI installed and authenticated. Use them only on machines where gh is set up.

13 — Lifecycle close

Completion & cleanup

When a molecule finishes, choose how to retire its children while preserving the right amount of evidence.

Distill — capture an ad-hoc epic as a formula

bd mol distill <epic-id> [formula-name] --var key=value

Output is written to the first writable formula search path. Variable detection uses word-boundary regex to avoid substring corruption.

Squash — collapse children into a single digest

bd mol squash <mol-id>                    # default body
bd mol squash <mol-id> --summary "AI summary"
bd mol squash <mol-id> --keep-children
  • Creates a closed task titled Digest: <root title> as a parent-child of the root.
  • Deletes ephemeral children by default; --keep-children retains them.
  • Auto-closes the root if it was a wisp and clears its Wisp flag.
  • All operations in a single transaction.

Burn — hard delete

bd mol burn <id...> --force         # no digest
bd mol burn <id...> --dry-run       # cascade preview

Wisp burn deletes via batch delete; persistent-molecule burn deletes the subgraph. No tombstones; no undo.

Promote — make a wisp durable

bd promote <wisp-id>

Copies the wisp and its labels/deps/events/comments into the durable tables. Preserves the original ID.

Wisp GC — sweep stale wisps

bd mol wisp gc                # 1h default, open wisps only
bd mol wisp gc --age 24h
bd mol wisp gc --all
bd mol wisp gc --closed --force    # purge closed regardless of age
bd mol wisp gc --exclude-type=agent,role,message

Stale discovery

bd mol stale                    # complete-but-unclosed molecule roots
bd mol stale --blocking          # roots with downstream blockers
bd mol stale --unassigned
bd stale                         # general stale issues
Decision rule

Use squash for disposable wisp work with an explicit summary. Use promote + acceptance-aware manual close for evidence-bearing work. Use burn only for true scratch.

14 — Operations

Operations & safety

Configuration, doctor, backup, history compaction, and the v1.1.0 upgrade recipe.

Configuration

bd config set no-push true
bd config set export.path beads.jsonl
bd config set import.orphan_handling <mode>
bd config show

Most keys also have an env-var equivalent (e.g., BD_NO_PUSH) and a CLI flag (e.g., --no-push). Env > flag > config file.

Doctor

bd doctor                       # general health
bd doctor --json                # machine-readable

Includes a migration-content-skew check that compares the local schema_migrations hashes against the cached remote-tracking ref (no network fetch).

Backup & restore

bd backup init /path/to/backup
bd backup sync
bd backup restore --force /path/to/backup

A full Dolt backup — not just an issues-table export — so it includes branches, history, working-set state, and non-issue tables.

History compaction

bd compact
bd flatten

Squashes Dolt history. Use when the database has grown large and you no longer need fine-grained history.

v1.1.0 upgrade recipe

Releases can carry schema migrations. A database that syncs to a Dolt remote must be migrated by exactly one designated clone.

  1. Back up first: bd export --all for portability + bd backup init /path && bd backup sync for a full database backup.
  2. Pick one designated clone to migrate. Run bd dolt pull, then start the new bd. The remote-migrate gate is on by default in v1.1.0.
  3. Other clones run bd bootstrap (or are auto-bootstrapped if origin advertises refs/dolt/data) to adopt the migrated state.
  4. Run bd doctor on every clone to surface migration-content skew.
  5. Migration 0052 introduces composite indexes for bd stale / bd ready: expect a one-time 10–25 s pause on rigs with >10K issues.

Safety checklist

  • Never use a JSONL diff as the cross-machine sync protocol.
  • Never run bd import during normal operation.
  • Never use bd edit in agents; use bd update flags.
  • Always pass --json in agent scripts.
  • Always pin v1.1.0 evidence when writing workflows.
  • Always back up before upgrading.
