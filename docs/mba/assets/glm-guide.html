1 Purpose & mental model

bd records work as issues (called beads) and connects them with dependencies. A human or agent files a piece of work, somebody (or some agent) claims it, does it, and closes it. Every change lands in a versioned database so you can review history, sync across machines, and audit decisions later.

Users

File work, prioritise, accept outcomes, approve gates.

Agents

Discover bd ready, claim, do, comment, close, hand off.

Orchestrators

Drive patrols, dispatch molecules, evaluate gates.

Storage

Local Dolt database under .beads/dolt/.

System boundary

Beads owns issue tracking primitives — the data model, dependency graph, history, sync, and gates. It deliberately does not own orchestration policy: there is no built-in runtime that executes a molecule for you. External agents and orchestrators consume the work graph through CLI commands.

First commands

# Project snapshot
bd status
bd info
bd quickstart

# Daily work pump
bd ready                # open issues with no active blockers
bd ready --claim --json # atomically claim the next ready issue

Source: docs/CLI_REFERENCE.md, docs/QUICKSTART.md, docs/PROJECT_CHARTER.md, docs/ARCHITECTURE.md.

2 Storage & truth

Memorise this three-line model — every sync decision in Beads follows from it.

1. Dolt is the source of truth. Issues, dependencies, labels, comments, events all live in the local Dolt database under .beads/dolt/. Each write auto-commits.
2. .beads/issues.jsonl is a passive export. Used by viewers, interchange, and migration. It is not the sync channel.
3. Sync lives under refs/dolt/data, separate from source Git under refs/heads/*. They share a remote URL but never collide.

Stores and movement

StoreWhat lives thereSyncs?Move command
.beads/dolt/ (Dolt)Issues, dependencies, labels, events, comments — the canonical recordYes, via Dolt remotebd dolt push / bd dolt pull
Wisp tables (wisps, wisp_*)Ephemeral issues (Ephemeral=true) — patrol loops, health checks, error tracesNo — dolt_ignored, local only(none — they never leave the machine)
.beads/issues.jsonlPassive export for viewers, migration, interoperabilityOnly if you commit it to Git (advisory)bd export / bd import
.beads/backup/Dolt-native backup for off-machine recoveryManualbd backup sync / bd backup restore
Source Git (your repo)Code, AGENTS.md, optional config filesYes, via Gitgit push / git pull
Anti-patterns. Do not run routine bd import .beads/issues.jsonl as a substitute for bd dolt pull. JSONL import is upsert-only and cannot infer records that were deleted, pruned, or never exported. Do not treat issues.jsonl as the wire protocol.

3 Install & repository setup

Beads is installed system-wide (not cloned into your project). The .beads/ directory inside a project only contains the issue database.

Install the CLI

# macOS / Linux — recommended
brew install beads

# Windows (PowerShell)
irm https://raw.githubusercontent.com/gastownhall/beads/main/install.ps1 | iex

# Cross-platform (macOS / Linux / FreeBSD)
curl -fsSL https://raw.githubusercontent.com/gastownhall/beads/main/scripts/install.sh | bash

# Node.js ecosystem
npm install -g @beads/bd

# Verify
bd version
bd help

Two embedded modes

Beads ships with an embedded Dolt engine by default — no external server required. Two go install build modes are relevant if you build from source:

  • Embedded-capable (cgo): CGO_ENABLED=1 GOFLAGS=-tags=gms_pure_go go install github.com/steveyegge/beads/cmd/bd@latest — full embedded Dolt; bd init just works.
  • Server-mode only (nocgo): CGO_ENABLED=0 go install github.com/steveyegge/beads/cmd/bd@latest — requires an external dolt sql-server and bd init --server.

Homebrew and the install script already ship the embedded-capable build.

Set up a repository

# From inside your project
bd init --quiet          # creates .beads/ and a Dolt database
bd setup claude          # or: codex, cursor, copilot, gemini, aider, mux, factory, junie, windsurf

# On a fresh clone (or after a move to a new machine)
bd bootstrap             # non-destructive: clones Dolt history, imports JSONL if needed, never deletes issues

# Health check
bd doctor                # start here — schema, migrations, hooks, file permissions, cycles
Safe path. bd bootstrap auto-detects the right action: clone from sync.remote, clone from git refs/dolt/data, restore from .beads/backup/, import from .beads/issues.jsonl, create a fresh database, or validate an existing one. It never destroys data.

Source: docs/INSTALLING.md, docs/SETUP.md, docs/QUICKSTART.md, docs/TROUBLESHOOTING.md, docs/UNINSTALLING.md.

4 Issue model

An issue is the central record. It has a stable content-hash ID (e.g. bd-a3f8e9), a title, a description, a status, a priority, a type, optional labels, comments, and metadata.

Built-in types

bug · feature · task · epic · chore · decision. Additional types can be enabled with types.custom in .beads/config.yaml or auto-registered by formulas.

5 Daily lifecycle

The normal flow is a tight loop around four verbs: create → claim → update → close. Reopen or defer when the situation changes.

bd create→ bd update --claim→ bd update -s in_progress→ bd comment / bd note→ bd close --reason→ bd ready

Create

bd create "Add OAuth login" -t feature -p 1 --assignee alice \
  --description "Implement login flow per spec" \
  --deps discovered-from:bd-7 \
  --json

Claim & update

bd update bd-abc --claim            # atomic: sets assignee + in_progress
bd update bd-abc --status in_progress --add-label backend
bd update bd-abc --append-notes "Hit issue with token refresh"
bd comment bd-abc "Investigating spec section 3"
bd note bd-abc "Workaround found"

Close & reopen

bd close bd-abc --reason "Implemented; tests green"
bd close bd-abc --continue --suggest-next   # auto-advance to next molecule step
bd close bd-abc --claim-next                # atomically claim the next ready issue
bd reopen bd-abc

Defer & undefer

bd defer bd-abc --until=tomorrow --reason="Waiting on API access"
bd undefer bd-abc

Hand-off patterns

  • Agent hand-off: bd update <id> --claim is idempotent if already claimed by the same actor; multiple agents racing to claim will see one winner.
  • After close: --suggest-next shows newly unblocked issues; --claim-next takes the next one atomically.
  • Multi-issue close: pass multiple IDs and one --reason per ID, positionally.

Source: docs/CLI_REFERENCE.md (bd create, bd update, bd close, bd reopen, bd defer, bd undefer, bd comment, bd note), docs/QUICKSTART.md.

6 Graph & ready work

Beads treats dependencies as first-class. Edges determine what is ready, what is blocked, and how molecules compose.

Dependency types

TypeBlocking?Affects ready work?Typical use
blocksYesYesSequential work — B cannot start until A closes.
conditional-blocksYesYesError path — B runs only if A fails.
waits-forYesYesFanout gate — B waits on dynamic children of a step.
parent-childStructural (not a hard blocker)YesHierarchy — an open parent keeps children unready; not rendered as a hard blocker.
related, relates-to, duplicates, supersedes, discovered-from, authored-by, tracks, until, caused-by, validates, delegated-from, approved-by, attestsNoNoCross-referencing, audit, attestation.

Readiness rules

bd ready uses GetReadyWork: open issues with no active blocking edge. It excludes in_progress, blocked, deferred, and hooked issues by default. bd blocked shows the inverse — open issues with at least one active blocker.

Commands

# Wiring
bd dep add bd-2 bd-1                         # bd-2 depends on bd-1 (blocks)
bd dep add bd-2 --blocked-by bd-1            # equivalent flag form
bd dep add bd-3 bd-2 --type conditional-blocks
bd dep bd-1 --blocks bd-2                    # shorthand (blocker first)
bd dep relate bd-abc bd-xyz                  # bidirectional relates_to

# Reading
bd dep list bd-x                             # what bd-x depends on
bd dep list bd-x --direction=up              # what depends on bd-x
bd dep tree bd-x --direction=both --max-depth=3
bd dep tree bd-x --format=mermaid            # export to Mermaid.js
bd dep cycles                                # detect cycles in the whole graph

# Ready work
bd ready --explain                           # why each issue is ready or blocked
bd ready --mol bd-patrol                     # steps within one molecule
bd ready --gated                             # molecules resumable because a gate closed

# Cleanup
bd dep remove bd-2 bd-1
bd dep unrelate bd-abc bd-xyz
Epics and hierarchy. An epic is a parent issue with children (added via --parent). Closing a parent does not cascade-close children, and a child can be opened independently of its parent. Use bd epic status and bd epic close-eligible to find epics where all children are now closed.

Source: docs/CLI_REFERENCE.md (bd dep, bd ready, bd blocked, bd children, bd epic, bd graph), docs/DEPENDENCIES.md, docs/graph-links.md, internal/types/types.go.

7 Find & understand work

Read/query commands share a common filter grammar (status, priority, labels, assignee, dates, metadata).

Find work to do

bd ready                                # claimable, no active blockers
bd ready -u                              # only unassigned ready issues
bd ready -p 1 --label backend            # priority-1 backend work
bd ready --mol bd-patrol                 # steps within a specific molecule

List & filter

bd list                                 # open issues, tree format
bd list --status open,in_progress --sort priority --limit 50
bd list --ready --assignee alice
bd list --label-any backend,infra --created-after 2025-01-01
bd list --include-gates                 # gates are hidden by default
bd list --include-templates             # template molecules hidden by default
bd list --metadata-field team=platform

Search & query

bd search "authentication" --status all
bd search "bd-5q"                       # fast prefix match on IDs
bd search "bug" --sort created --reverse
bd query                                # simple query language; see bd query --help

Understand a specific issue

bd show bd-abc --long                  # all fields, extended metadata
bd show bd-abc --thread                 # full conversation thread (for messages)
bd show bd-abc --refs                   # reverse lookup of references
bd show bd-abc --as-of <commit>         # time travel (requires Dolt)
bd history bd-abc --limit 10            # all commits that touched bd-abc

Reports & hygiene

8 Collaboration & sync

Cross-machine collaboration uses Dolt remotes. For Git-hosted projects the remote can be the same origin URL — Beads writes issue history under refs/dolt/data, fully separate from source branches under refs/heads/*.

Safe order

# Before starting work
bd dolt pull

# Do your work (create, update, close, ...)

# To share with teammates
bd dolt push
Why bd dolt pull first? Dolt history merges cleanly because it is cell-level. Pulling before you write minimises divergence and merge work. The bd vc conflicts / bd vc resolve commands surface any unresolved Dolt merges.

Configure the Dolt remote

bd dolt remote list
bd dolt remote add origin <git-or-dolt-url>
bd dolt remote remove origin

# Common URL forms
#   git+ssh://git@github.com/org/repo.git
#   git+https://github.com/org/repo.git
#   https://doltremoteapi.dolthub.com/<user>/<repo>

Fresh clone / recovery

bd bootstrap              # auto-detect: clone from sync.remote, git refs/dolt/data, backup, or JSONL
bd bootstrap --dry-run    # preview the plan
bd bootstrap --json       # machine-readable plan

Repair (older projects without auto-wired origin)

# On the machine with the authoritative local DB:
bd export -o .beads/issues.pre-remote.jsonl   # optional audit export
bd dolt remote add origin <git-or-dolt-url>
bd dolt push

# Other machines:
bd dolt pull
# or, if the local DB is stale or missing:
bd bootstrap

Worktrees & multi-repo

bd worktree create                       # parallel development on one clone
bd worktree list
bd worktree remove

bd repo add <path-or-url>                # track an additional repository in one workspace
bd repo list
bd repo sync                             # manually trigger multi-repo sync

Conflicts & version control

bd vc status                             # current branch + uncommitted changes
bd vc commit -m "msg"                    # commit all staged changes
bd vc merge <branch>                     # merge a Dolt branch
bd recompute-blocked                     # repair stale is_blocked flags after a pull
Wisps never sync. Wisp tables are dolt_ignored. An agent that runs on multiple machines must accept that wisp state is per-machine. Durable audit history must come from a poured molecule, a separate record outside bd, or from bd promote <wisp-id>.

Source: docs/SYNC_CONCEPTS.md, docs/SYNC_SETUP.md, docs/MULTI_REPO_AGENTS.md, docs/MULTI_REPO_MIGRATION.md, docs/WORKTREES.md, docs/CLI_REFERENCE.md (bd dolt *, bd vc *, bd bootstrap, bd worktree *, bd repo *).

9 Agent workflow & memory

Beads is designed for human–agent teams. Agents discover work, claim atomically, do the work, narrate progress in comments, and close with a reason. Persistent memory survives context compaction.

Prime (context recovery)

bd prime                 # ~1–2k tokens of workflow context; adapts to MCP vs CLI mode
bd prime --memories-only # for hook contexts that only need memories
bd prime --hook-json     # wrap output in a SessionStart hook JSON envelope

Install editor/agent integrations to auto-inject bd prime on session start:

bd setup --list                              # see all recipes
bd setup claude                              # Claude Code: SessionStart/PreCompact hooks
bd setup codex                               # Codex: skill + AGENTS.md + native /hooks
bd setup cursor                              # Cursor IDE: .cursor/rules/beads.mdc
bd setup copilot                             # Copilot CLI: plugin + repo instructions
bd setup gemini                              # Gemini CLI integration
bd setup aider                               # Aider config
bd setup claude --check                      # verify installation status
bd onboard                                   # minimal snippet for unsupported agents

Comments (attribution & progress)

bd comment bd-abc "Implemented per spec section 3"
bd comment bd-abc --file notes.txt
echo "piped comment" | bd comment bd-abc --stdin
bd comments bd-abc --json                    # list with full bodies

Persistent memory

bd remember "Dolt phantom DBs hide in three places" --key dolt-phantoms
bd remember "always run tests with -race flag"
bd recall dolt-phantoms
bd memories                                  # list or search all memories
bd forget dolt-phantoms                      # remove a memory

Memories are injected at bd prime time, so every session starts with them — no manual loading. Place a .beads/PRIME.md file in the local clone to override the default prime output entirely.

Standard agent loop

bd prime→ bd ready→ bd update --claim→ bd comment→ bd close --reason→ bd ready --claim-next

Audit trail

Every write produces a Dolt commit and an event row. For agent tool-call logging, append entries to .beads/interactions.jsonl:

bd audit record --kind tool_call --tool-name bd --issue-id bd-abc
bd audit label <entry-id> --label good --reason "clean handoff"

10 Formula system advanced

A formula is a TOML or JSON file that describes a reusable workflow as a DAG of steps with optional variables, inheritance, composition rules, and aspect hooks. Formulas are the highest-level source layer in the beads chemistry metaphor.

Anatomy

ElementPurpose
formula, description, versionIdentity and human description.
typeOne of workflow, expansion, aspect, convoy.
varsNamed placeholders with optional default, required, enum, pattern. Pattern: {{name}}.
stepsOrdered list. Each step has id, title, optional depends_on/needs, waits_for, condition, gate, loop, children, assignee, labels.
extendsInheritance chain. Child overrides parent steps by ID; child-only steps appended; vars/compose merged. Cycle-detected.
compose.bond_pointsNamed attachment sites (authoring metadata in v1.1.0 — not yet addressable from CLI flags).
compose.expand / compose.mapReplace target step(s) with an expansion formula's template. Max depth 5.
compose.aspectsAspect formulas applied after expansions.
advicePer-formula before/after/around insertions, keyed by glob. Self-matching prevented.
phase (liquid or vapor)Recommends bd mol pour or bd mol wisp. Pouring a vapor-phase formula produces a warning.

Search paths (priority order)

  1. <resolved-beads-dir>/formulas/ (active project)
  2. <checkout-root>/.beads/formulas/ (repo-local)
  3. ~/.beads/formulas/ (user)
  4. $GT_ROOT/.beads/formulas/ (shared workspace root, if GT_ROOT set)

Earlier paths shadow later ones for the same formula name.

Discovery & inspection

bd formula list                          # all formulas from all search paths
bd formula list --type workflow
bd formula show mol-feature              # vars, steps, advice, compose, pointcuts
bd formula show security-audit --json
bd formula convert --all                 # migrate JSON → TOML (preferred)
bd mol seed mol-feature                  # verify a formula is accessible and cookable

Compile vs runtime

bd cook mol-feature                      # compile mode (default): keeps {{vars}} intact
bd cook mol-feature --var name=auth      # runtime mode: substitutes vars (requires all)
bd cook mol-feature --dry-run            # preview without writing
bd cook mol-feature --persist            # LEGACY: writes proto to DB; modern flow keeps protos ephemeral

Conditional steps

  • Cook-time condition: {{var}}, !{{var}}, {{var}} == value, {{var}} != value. Truthy: anything except "", "false", "0", "no", "off".
  • Runtime conditions (gate labels): step.status == 'complete', aggregates children(...).all|any|count(...), external file.exists('path') / env.NAME op value. Aggregate all over an empty set returns false.
Validation. Formula.Validate() rejects: empty names, invalid versions/types, required + default on the same var, duplicate step IDs, missing titles, invalid priority, dangling depends_on/needs/waits_for, invalid gate/condition/on_complete, inconsistent bond points (both anchors set).

Source: docs/CLI_REFERENCE.md (bd formula, bd cook, bd mol seed), internal/formula/types.go, internal/formula/parser.go, internal/formula/stepcondition.go, internal/formula/condition.go, internal/formula/expand.go, internal/formula/advice.go, internal/formula/controlflow.go, cmd/bd/cook.go, cmd/bd/formula.go.

11 Workflow instances advanced

Formulas become real work through three phase transitions. The chemistry metaphor: proto (solid template) → molecule (poured liquid, durable) or wisp (vaporised, ephemeral).

Proto (solid)

A proto is a template epic labelled template. In v1.1.0 protos are normally ephemeral: bd mol pour / bd mol wisp cook the formula inline at use time, and nothing is written to the database for the proto itself. The legacy bd cook --persist writes a persisted proto bead — the modern flow avoids this.

Molecule (liquid)

A molecule is an instantiated DAG of beads — a poured proto plus its parent-child and dependency edges. The molecule root is an epic of type molecule. Children attach via parent-child; steps connect via blocks (sequential), conditional-blocks (error path), or waits-for (fanout gate).

bd mol pour mol-feature --var name=auth
bd mol pour mol-review --var pr=123 --assignee alice
bd mol pour mol-feature --attach mol-security --attach-type parallel
bd mol show bd-patrol --parallel          # which steps can run concurrently
bd mol progress bd-patrol                 # completed/total, rate, ETA
bd mol current                            # where am I in my active molecule?
bd mol last-activity bd-patrol            # most recent change timestamp
bd mol stale                              # complete-but-unclosed molecule roots

Wisp (vapor)

A wisp is an ephemeral issue (Ephemeral=true). Wisp rows live in dolt_ignored tables (wisps, wisp_labels, wisp_dependencies, wisp_events, wisp_comments) — they never enter Dolt history and never sync. Wisp IDs look like bd-wisp-x7k.

bd mol wisp mol-patrol --var target=db
bd mol wisp list                          # all wisps (old = not updated in 24h)
bd mol wisp list --all                    # include closed wisps
bd mol wisp gc                            # preview: abandoned open wisps > 1h old
bd mol wisp gc --age 24h --all
bd mol wisp gc --closed --force           # purge ALL closed wisps
bd mol wisp gc --exclude-type agent,rig   # protect infra wisps

Wisp categorical type (WispType): heartbeat · ping · patrol · gc_report · recovery · error · escalation. Infra types (agent, role, message, …) are protected from GC by default.

Phase transitions

Proto → Molecule

bd mol pour <formula|proto> — cook (if formula) → clone subgraph → atomic transaction.

Proto → Wisp

bd mol wisp <formula|proto> — same flow with Ephemeral=true; routed to wisp tables.

Wisp → Durable

bd promote <wisp-id> — atomic copy from wisp tables to permanent tables; inbound deps retargeted.

Wisp → Digest

bd mol squash <mol> — replace ephemeral children with one permanent digest issue.

Source: docs/CLI_REFERENCE.md (bd mol *, bd promote), docs/MOLECULES.md, docs/ARCHITECTURE.md, internal/storage/dolt/wisps.go, internal/storage/dolt/ephemeral_routing.go, cmd/bd/pour.go, cmd/bd/wisp.go, cmd/bd/template.go.

12 Bonds & gates advanced

Bonds

A bond is the typed relationship created by bd mol bond. The command is polymorphic across operands:

ABOutcome
formula + formulaformula + proto / proto + protoCook (if needed) → compound proto with template label and parent-child edges to both.
formula/proto + molecule—Cook/load the proto → spawn via cloneSubgraph → atomic attach in the same transaction.
molecule + molecule—Add a blocking dep between them. Cycle-checked via BFS.

Bond types map to dependency types:

  • sequential (default) → blocks
  • parallel → parent-child
  • conditional → conditional-blocks
bd mol bond mol-feature mol-deploy                     # compound proto
bd mol bond mol-feature mol-deploy --type parallel
bd mol bond mol-feature bd-abc                          # attach proto to molecule
bd mol bond bd-abc bd-def                               # join two molecules
bd mol bond mol-critical-bug wisp-patrol --pour         # persist found bug
bd mol bond mol-arm bd-patrol --ref arm-{{name}} --var name=ace   # dynamic ID: bd-patrol.arm-ace
Bond points in v1.1.0. Formulas can declare compose.bond_points (named after_step/before_step anchors with a parallel flag), but they are authoring metadata only — the bd mol bond flag set does not expose them yet.

Gates

A gate is an issue of type gate that blocks another issue until its AwaitType condition is satisfied. The gate ID convention is <parent>.gate-<step-id> when created from a formula step.

Gate typeResolved byEscalated by
humanManual bd close <gate-id> or bd gate resolve <gate-id>—
timerbd gate check auto-closes once created_at + timeout has passed—
gh:runbd gate check sees status=completed AND conclusion=success via the gh CLIconclusion ∈ {failure, canceled}
gh:prbd gate check sees state=MERGEDstate=CLOSED
beadDisabled in v1.1.0 — multi-rig routing removed. Always reports "cannot be checked".—
mailType is documented but not implemented in bd gate check at v1.1.0.—
 
bd gate create --type=human --blocks bd-9 --reason="Need design review"
bd gate create --type=timer --blocks bd-9 --timeout=2h
bd gate create --type=gh:pr --blocks bd-9 --await-id=42
bd gate check                                # all open gates
bd gate check --type=gh:run --escalate
bd gate discover                             # match gh:run gates to recent workflow runs
bd gate list --all
bd gate resolve bd-x-gate-step               # alias for bd close
bd gate add-waiter bd-x-gate-step <addr>     # register an agent for wake notification

# Resume molecules after a gate closes
bd ready --gated
bd mol ready --gated --json
Default limit. bd gate check --limit defaults to 100. No hard cap is enforced in the check path; passing a higher value iterates that many gates per call.

13 Completion & cleanup advanced

v1.1.0 release theme — safe schema-migration and upgrade path

  • The state-aware remote-migrate gate is on by default; bd refuses to silently migrate a remote-backed database.
  • v53 migration no longer traps the database; covers wisp_dependencies split-column drift and adds a recovery gate that lets an already-failed-v53 database self-heal on next open.
  • Dirty working set no longer deadlocks migration recovery in embedded mode; bd dolt commit / bd vc commit open past the guard.
  • bd doctor compares local schema_migrations content hashes against the cached remote-tracking ref (no network fetch) and surfaces silent schema forks as advisories.
  • bd bootstrap adopts a remote's already-migrated state for cross-clone recovery.

Integrations

Two-way sync is available for external trackers via dedicated subcommands (status, pull, push, sync):

bd jira · bd linear · bd github · bd gitlab · bd notion · bd ado (Azure DevOps).

Key cautions

  1. Never run routine bd import .beads/issues.jsonl as a substitute for bd dolt pull.
  2. Never use bd edit in agent contexts (it opens $EDITOR); use bd update with flags instead.
  3. When upgrading a remote-backed database across a schema migration, follow the ordered upgrade recipe; replacing the binary alone is not enough.
  4. bd flatten, bd mol burn, bd delete --cascade, and bd admin reset are irreversible.
  5. Wisps never sync. Plan durable audit history accordingly.
  6. Always use non-interactive forms (-f, --force, --yes) in scripts to avoid hangs on confirmation prompts.
